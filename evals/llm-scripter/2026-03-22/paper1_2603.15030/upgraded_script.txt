The section we are discussing focuses on the evaluation metrics used in the research study. The primary metric employed is called the Average Pass Rate, abbreviated as APR. This metric represents the proportion of questions that are correctly answered by the models being evaluated.

To delve into how tools are used with more detail, the concept of the "Effective Toolchain" is introduced. This refers to the minimal sequence of tool calls required to produce the correct final answer. Determining this sequence involves a backtracking process from the final output all the way back to the original input image.

In addition to APR, the researchers evaluate the models using three more specific metrics. The first of these is the Tool Call Rate, or TCR, which measures the proportion of tasks in which a model uses at least one tool. Next, the Mean Absolute Error, abbreviated MAE, quantifies how much the predicted toolchain lengths deviate from their true lengths. Lastly, there is Tool Usage Efficiency, represented as Eff-tool. This metric assesses the precision and conciseness of sequences of tool calls by comparing the number of effective steps with the total number of predicted steps.

The Mean Absolute Error and Eff-tool are mathematically described as follows. The Mean Absolute Error is calculated as the average of the absolute differences between the lengths of the ground-truth toolchains and the total toolchains for all evaluated samples. The formula states: MAE equals one over N, multiplied by the sum of the absolute differences between L_G,i and L_T,i. The Eff-tool is computed by dividing the sum of effective toolchain lengths, denoted as L_e,i, by the sum of the total toolchain lengths, denoted as L_T,i. Here, N represents the total number of evaluated samples, and the variables L_G,i, L_T,i, and L_e,i denote the lengths of the ground-truth toolchain, total toolchain, and effective toolchain, respectively, for each sample indexed by i.

In this section, titled "Experiment Setup", we go through the comprehensive details of the model selection and evaluation framework used in the study.

Starting with the model selection, the researchers evaluated nineteen mainstream multi-language capable models, known as MLLMs. These models include both proprietary and open-source models with various backgrounds in tool-use training. The authors have grouped these models into four distinct categories.

The first category consists of proprietary tool-use models. These are closed-source systems that possess robust native capabilities. Within this group are models such as GPT-o3, and GPT-o4-mini.

The second category is made up of leading proprietary general-purpose models. Examples in this category include GPT-5.2, Gemini-3.0-Pro, Gemini-3.0-Flash, Gemini-2.5-Pro, and GPT-4o.

The third group transitions into the open-source domain and is comprised of dedicated open-source tool-use models. This includes specialized architectures such as DeepEyes, Thyme, DeepEyesV2, and V-Thinker.

Finally, for open-source general-purpose models, the researchers conducted evaluations on both the Instruct and Thinking variants of Qwen3-VL models. These evaluations were carried out across an extensive range of parameter scales, ranging from 8 billion to 235 billion parameters.

Moving on to the evaluation framework, to measure the benefits of tool invocation, the authors compared direct answering baselines against tool-augmented reasoning. This was done across both interface-driven and code-driven paradigms. The comparison aimed to reveal how a model's programming proficiency affects the overall orchestration success.

The evaluation workflow is visually represented in Figure 1 of the paper. To ensure a consistent environment, all model evaluations were accessed through standardized APIs. However, to maintain fairness, the orchestration strategy was adapted based on each model's inherent capabilities.

Specifically, the researchers used the Qwen-Agent framework to manage both code and interface implementations for models with strong native tool-calling support.

Conversely, for models with limited native tool-calling proficiency, which applied primarily to open-source tool-use models, the Thyme framework was employed. This approach facilitated code generation. To aid this process, detailed tool descriptions were provided, explicitly outlining usage instructions and parameter specifications, ensuring that even models lacking specialized OpenCV training could understand and invoke the available interfaces.

A comprehensive dual-stream evaluation protocol was also implemented. This involved integrating a deterministic rule-based matching system with a large language model as a judge, specifically using the GPT-4o model as a judge paradigm. To conclude the evaluation setup, it’s mentioned that they utilized a framework called VLMEvalKit. This framework enabled the rigorous quantification of tool-use performance. Given the dataset consisted of single-choice and open-ended tasks, a dual-stream evaluation protocol was executed. The first stream used a deterministic rule-based method, employing regular expressions to extract the output for comparison against the Ground Truth. This process utilized both an exact match and math verification to ensure accuracy. The second stream employed a large language model as a judgment system, using GPT-4o-mini. Our benchmark presents a significant challenge to existing large multi-modal models. These results clearly indicate that there is substantial room for improvement in complex tool-augmented reasoning.

Next, we found that simple interfaces outperform complex code generation for models with limited tool-calling proficiency. Interestingly, specialized open-source models demonstrate superior performance when using code-based execution. Their performance with code consistently exceeds that of interface-based methods.

Furthermore, the proposed set of 32 tools provides comprehensive coverage for diverse visual scenarios. Both open-source and proprietary models achieve superior or comparable performance when using the interface-based approach. This finding suggests that our toolset effectively supports various task requirements across different architectures.

To delve into specific challenges, the benchmark presents a significant hurdle to current multi-modal large language models. In our base setting, performance is relatively low, with scores ranging from approximately 22 percent to around 46 and a half percent. Now let's compare proprietary models with open-source models. Our evaluation highlights a pronounced performance disparity between the two, especially in leveraging external tools. Proprietary models not only establish robust baselines but also exhibit substantial performance gains when augmented with tool-use capabilities. When we examine the performance based on task analysis, we categorize them into three major classes: models performing coarse-grained tool invocation, which are foundational for tool use; quantitative visual estimation with fine-grained manipulations; and compositional visual reasoning tasks.

On the contrary, in quantitative visual estimation, outcomes are split. There is a noticeable degradation in high-precision perception tasks. Finally, in compositional visual reasoning tasks, models achieve remarkable gains, such as Gemini-3.0-Pro's Spatial Reasoning score, which improves by 17.78 percent.
